# AnoClaw 用户指南

> 面向第一次使用 AnoClaw 或想更好协作的用户。

## AnoClaw 是什么

AnoClaw 是一个本地桌面 AI 平台。你可以和主 agent 对话，也可以让它组织团队、调用工具、读写项目文件、开发插件、维护记忆，并通过 WebSocket 实时展示思考、工具调用和任务进展。

核心能力：

- 多 agent 协作：MainAgent、Manager、Member、SubAgent。
- 本地工作区操作：读文件、写文件、搜索、运行命令。
- 插件系统：给 agent 增加工具、页面、API 路由和专用能力。
- Skills：把复杂工作流封装成可调用能力。
- Memory：保存用户偏好、项目事实和长期团队经验。
- Docs：随包携带的通用知识库，供 agent 查询产品、插件、API、设计和排障资料。

## 怎么提需求

最有效的请求通常包含：

```text
目标：我想完成什么
范围：涉及哪些文件、插件、页面或业务
约束：不要改什么、必须兼容什么
验收：怎样算完成
偏好：语言、风格、输出形式
```

示例：

```text
帮我做一个插件，读取 workspace 里的 csv 并生成汇总报告。
要求：
- 插件名 csv-inspector
- 提供一个 agent 工具和一个前端页面
- 页面使用 AnoClaw tokens，不要引入 React
- 做完后告诉我怎么验证
```

## 会话与工作区

每个会话都有上下文。agent 会根据当前会话、当前 workspace、已启用工具、skills 和 memory 做决定。

建议：

- 一个主题尽量放在同一个会话里，方便保留上下文。
- 大任务可以让 agent 先列计划，再执行。
- 如果切换项目或目录，明确告诉 agent 新的 workspace。
- 不要把密钥直接发给 agent，优先使用设置页或本地配置。

## 多 agent 什么时候有用

适合多 agent：

- 大型代码改造，需要并行阅读不同模块。
- 插件开发同时涉及后端、前端、文档、测试。
- 需要一个 agent 执行，一个 agent 复核。
- 任务可以拆成互不阻塞的子任务。

不一定需要多 agent：

- 单文件小改动。
- 简短解释。
- 只需要运行一个命令。
- 需求还没明确，先和主 agent 聊清楚更快。

## Skills、Memory、Docs 怎么理解

| 能力 | 你可以怎么用 |
|---|---|
| Skills | “用某个流程来做这件事”，比如测试、打包、文档生成 |
| Memory | “以后记住我的偏好/这个项目事实” |
| Docs | “查 AnoClaw 的插件/API/设计/排障资料” |

如果你说“记住这个”，agent 应使用 memory。如果你说“把这套通用规则写进知识库”，agent 应更新 docs。如果你说“以后遇到这类任务按这个流程做”，更适合创建或更新 skill。

## 插件能做什么

插件可以：

- 注册 agent 工具。
- 添加 UI 页面。
- 注册 HTTP API。
- 调用系统 LLM。
- 搜索/读写 workspace 文件。
- 订阅系统事件。
- 往 UI 插槽挂载内容。
- 注入短提示词规则。
- 高级情况下覆盖内核扩展点。

想开发插件时，让 agent 先读 `docs/plugin-dev.md` 和 `docs/plugin-api.md`。

## 常见工作流

### 修改代码

```text
请修复 X 问题。先定位原因，再做最小改动。完成后运行相关测试并总结。
```

### 开发插件

```text
请创建一个插件，功能是 X。需要工具、页面、配置项和调试说明。
```

### 排查故障

```text
AnoClaw 双击没反应。请按 docs/troubleshooting.md 检查，不要删除用户数据。
```

### 维护知识库

```text
把这次确认下来的插件热重载规则写进 docs，确保以后 agent 能查到。
```

## 高质量协作建议

- 明确告诉 agent “可以改代码”或“先只分析”。
- 如果结果要面向非技术用户，提前说明受众。
- 对 UI 任务给出风格参考；没有参考时让 agent 从 `docs/design-md/` 选择。
- 对高风险任务要求 agent 先说明验证方式。
- 如果 agent 答案和实际软件不一致，请指出，agent 应更新 docs 或 memory。
