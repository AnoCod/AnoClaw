# Agent Docs 调度与维护规则

> 面向 AnoClaw agent。目标是让 agent 智能使用和维护 `docs/`，但不把它误用成 skill 或 memory。

## 什么时候必须查 docs

遇到这些任务时，先查 `docs/`：

- 用户问 AnoClaw 功能、架构、插件、API、UI、设计预设、排障。
- 用户要 agent 创建、修改或调试插件。
- 用户要解释 agent 组织、多会话、memory、skills、docs 的关系。
- 用户要做可复用的产品能力、开发范式或操作手册。
- 回答依赖当前 AnoClaw 实现细节，而不是通用常识。

## 调度流程

```text
用户问题
  -> 判断是否属于 AnoClaw 产品/开发/设计/排障知识
  -> Read docs/README.md 或 docs/manifest.json
  -> Grep/Glob 定位候选资料
  -> Read 相关文档
  -> 必要时核对源码/配置/运行结果
  -> 回答、执行，或更新 docs
```

优先路径：

| 任务 | 入口 |
|---|---|
| 插件怎么写 | `docs/plugin-dev.md` |
| 某个插件 API 怎么调用 | `docs/plugin-api.md` |
| 插件页面怎么做 | `docs/plugin-ui-guide.md` |
| agent 如何协作 | `docs/agent-collaboration.md` |
| 平台架构和数据流 | `docs/architecture-overview.md` |
| 启动、插件、页面、构建问题 | `docs/troubleshooting.md` |
| 资料维护 | `docs/agent-docs.md` |
| 品牌风格 | `docs/design-md/<brand>/DESIGN.md` |

## docs、skills、memory 的边界

| 内容类型 | 应放哪里 | 原因 |
|---|---|---|
| 公共产品知识、API、操作指南、排障手册 | `docs/` | 所有用户和 agent 都能复用 |
| 需要按步骤执行的专业流程 | `skills/` | agent 需要显式调用流程能力 |
| 用户偏好、项目事实、团队约定 | `memory/` | 个性化、长期、可检索 |
| 当前任务状态、临时 todo | 当前会话 / task | 不应污染长期知识 |
| 密钥、token、私人信息 | 不写入 | 安全与隐私 |

判断口诀：

- “以后所有人都可能受益” -> `docs/`
- “agent 做这类事时需要按流程执行” -> `skills/`
- “只和这个用户、团队或项目有关” -> `memory/`
- “只和这次任务有关” -> 不持久化到 docs

## 什么时候更新 docs

满足任一条件即可更新：

- 用户明确要求补充或维护知识库。
- 发现 `docs/` 与源码或实际行为不一致。
- 新增/修改了可复用能力，未来 agent 需要知道。
- 排障过程中确认了稳定的原因和恢复步骤。
- 新增了面向用户的概念、限制、流程或最佳实践。

不应更新：

- 只是本次会话的临时进展。
- 尚未验证的猜测。
- 用户隐私、账号、路径中包含敏感信息。
- 需要经常变化的数据，例如当天新闻、价格、临时服务状态。

## 更新标准

更新前：

1. 用 `Grep` 找现有资料，避免重复。
2. 如果是 API/架构/行为，读取对应源码或运行命令验证。
3. 判断是否需要同步 `docs/README.md` 与 `docs/manifest.json`。

更新时：

- 标题清楚，第一段说明适用人群。
- 保持事实可验证，不写“可能”“应该”当成结论。
- 用表格列决策、路径、命令、症状。
- 示例要能直接复制运行，路径保持相对 AnoClaw 根目录或明确为绝对路径。
- 对用户可见文案避免内部黑话，必要时先解释概念。

更新后：

- `docs/README.md` 有新入口或相关链接。
- `docs/manifest.json` 有新条目、tags、summary。
- 如果改了代码或 prompt，运行构建/测试。
- 如果是重大变更，记录到工作日志。

## 回答规范

当基于 docs 回答时：

- 只引用与问题相关的最少资料。
- 如果 docs 与源码冲突，以源码/运行结果为准，并更新 docs。
- 不把 docs 的存在当成用户必须知道的负担；直接给答案即可。
- 如果问题更适合 skill 或 memory，说明边界并使用对应机制。

## 维护质量红线

- 不要复制大段源码到 docs，写稳定接口和行为即可。
- 不要把 docs 写成流水账。
- 不要用 docs 记录个人偏好。
- 不要让设计预设覆盖 AnoClaw 自身 UI 可用性原则。
- 不要为了“知识库更大”添加低密度内容。每份资料都要解决真实问题。
